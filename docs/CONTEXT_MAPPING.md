## Beat Profiles

TuneSoar supports five brainwave frequency bands:

| Band | Frequency Range | Default Hz | Effect | Risk Level |
|---|---|---|---|---|
| 🧠 **Gamma** | 30–40 Hz | 35 Hz | High cognition, peak awareness, information processing | ⚠️ **High** — disabled by default. Do not use if prone to seizures. |
| ⚡ **Beta** | 13–30 Hz | 15 Hz | Active focus, concentration, analytical thinking, problem-solving | Moderate |
| 🌊 **Alpha** | 8–13 Hz | 10 Hz | Relaxed alertness, calm focus, flow state, light meditation | Low |
| 🧘 **Theta** | 4–8 Hz | 6 Hz | Deep meditation, creativity, intuition, deep relaxation, light sleep | Low–Moderate |
| 🌙 **Delta** | 1–4 Hz | 2 Hz | Deep dreamless sleep, physical healing, pain relief | Moderate (drowsiness) |

Each profile is generated by playing a **carrier frequency** (default: 200 Hz) in one ear and **carrier + beat frequency** in the other. The brain perceives the difference as the beat.

Example: For a 15 Hz Beta beat with a 200 Hz carrier:
- **Left ear:** `200 − 7.5 = 192.5 Hz`
- **Right ear:** `200 + 7.5 = 207.5 Hz`
- **Perceived beat:** `207.5 − 192.5 = 15 Hz`

## Context Detection Priority

When TuneSoar detects a new context, it evaluates signals in this order:

```
   └─ Most specific signal. If a URL is available, it is checked first.
      YouTube has additional keyword-based sub-matching.

2. App Name (exact match)
   └─ The active window's process name / bundle name is matched against
      all known app names in the default + custom mappings.

3. App Name (fuzzy match)
   └─ If no exact match, a case-insensitive substring search is performed
      against both the app name and window title.

4. Time-Based Fallback
   └─ If no app or URL match, check the current local hour:
      - 22:00–05:59 → Check for reader/research apps in window title
        (Kindle, Reader, Books, Pocket, Instapaper) → SleepPrep
      - Otherwise → Ambient (default)

5. Ambient (Default)
   └─ If absolutely nothing matches → Ambient (Alpha 10 Hz)
```

### Priority Diagram

```
┌──────────────────────────────────────┐
│       Detection Cycle (every 3s)     │
└──────────────────┬───────────────────┘
                   ▼
         ┌─────────────────┐
         │ URL Available?   │──Yes──▶ URL Match → Context
         └────────┬─────────┘            (highest priority)
                  │ No
                  ▼
         ┌─────────────────┐
         │ Exact App Match? │──Yes──▶ App Context
         └────────┬─────────┘
                  │ No
                  ▼
         ┌─────────────────┐
         │ Fuzzy App Match? │──Yes──▶ App Context (fuzzy)
         └────────┬─────────┘
                  │ No
                  ▼
         ┌─────────────────┐
         │ Is it 22:00-06?  │──Yes──▶ Sleep Prep (if reading)
         └────────┬─────────┘          or Ambient (if not reading)
                  │ No
                  ▼
         ┌─────────────────┐
         │    Ambient       │  ← Default fallback
         └─────────────────┘
```

## Custom Mappings

Users can create custom context mappings from the **Context Mappings** settings page (Pro tier feature).

### Mapping Format

Each custom mapping consists of:

| Field | Type | Description |
|---|---|---|
| `pattern` | `string` | The app name or URL pattern to match |
| `pattern_type` | `"app"` or `"url"` | Whether this matches an app name or URL |
| `context_type` | `ContextType` | Which context to classify as (e.g., "Coding", "Writing") |
| `beat_type` | `BeatType` | Which beat profile to use (Delta, Theta, Alpha, Beta, Gamma) |
| `beat_frequency` | `float` | The exact beat frequency in Hz (must be within the band's range) |
| `enabled` | `boolean` | Whether this mapping is active |

### Example Custom Mappings

```json
{
  "pattern": "Twitter",
  "pattern_type": "app",
  "context_type": "Communication",
  "beat_type": "Beta",
  "beat_frequency": 13,
  "enabled": true
}
```

```json
{
  "pattern": "reddit.com",
  "pattern_type": "url",
  "context_type": "PassiveWatch",
  "beat_type": "Alpha",
  "beat_frequency": 10,
  "enabled": true
}
```

```json
{
  "pattern": "Anki",
  "pattern_type": "app",
  "context_type": "Coding",
  "beat_type": "Beta",
  "beat_frequency": 18,
  "enabled": true
}
```

### Custom Mapping Behavior

- Custom mappings **override** default mappings for the same pattern
- Custom mappings are stored in the local SQLite database (`context_mappings` table)
- Mappings can be **toggled on/off** without deleting them
- Duplicate `(pattern, pattern_type)` pairs are upserted (the latest write wins)
- Custom URL mappings follow the same **substring matching** rules as default URL mappings

## URL Pattern Matching Rules

### Browser Extension to Desktop App

3. The desktop app performs a **case-insensitive substring match** against all URL mappings
4. The first matching pattern wins (default mappings checked first, then custom mappings)

### Matching Algorithm

```rust
// Simplified algorithm (from context/detector.rs)
fn match_url(url: &str) -> Option<ContextType> {
    let url_lower = url.to_lowercase();

    // Special YouTube handling
    if url_lower.contains("youtube.com") {
        if contains_audio_keywords(&url_lower) {
            return Some(Music);  // auto-pause
        }
        return Some(PassiveWatch);
    }

    // Check all URL mappings (case-insensitive substring)
    for (pattern, context_type) in url_mappings.iter() {
        if url_lower.contains(&pattern.to_lowercase()) {
            return Some(*context_type);
        }
    }

    None
}
```

### Key Rules

- **Substring matching, not exact matching:** `youtube.com` matches `https://www.youtube.com/watch?v=...`
- **Case-insensitive:** `GitHub.com` matches `github.com`
- **First match wins:** If `discord.com` appears before `discord.com/app` as a mapping, the `discord.com` mapping applies to both
- **No wildcard or regex support** in default mappings (custom mappings are also substring-based)
- **Hostname only:** The extension strips everything after the hostname. The full URL path is not sent.

## Fuzzy App Matching

When no exact app name match is found, TuneSoar performs a **case-insensitive contains** search:

1. Convert the detected `app_name` to lowercase
2. Convert the `window_title` to lowercase
3. For each known app pattern:
   - Check if `app_name.contains(pattern_lowercase)` OR `window_title.contains(pattern_lowercase)`
   - Return the first match found

This means:
- `"Visual Studio Code"` matches because the app name literally contains `"Visual Studio Code"`
- A window titled `"project.py — Sublime Text"` matches the `"sublime text"` pattern even if the process name is something else
- `"google-chrome"` would NOT match `"chrome"` because the contains check works on the pattern side (not the detected name side), so `"google-chrome".contains("chrome")` IS true, and it would match if "Chrome" were a listed app name — but Chrome itself is not in the default mappings (context comes from the URL, not the browser app name)

## Idle Detection Logic

TuneSoar tracks user activity via the **last active timestamp**, updated on every context detection cycle.

### Idle Detection Parameters

| Parameter | Value | Configurable? |
|---|---|---|
| Detection cycle interval | Every 3 seconds (default, adjustable 1–30s in Settings) | Yes |
| Idle threshold | 5 minutes (300,000 ms) | No (hardcoded) |
| Action on idle | Beat frequency set to 0 Hz (fade out) | N/A |
| Action on wake | Resume last detected context's beat profile | N/A |

### How It Works

```rust
// From context/detector.rs
pub fn is_idle(&self) -> bool {
    let now = chrono::Utc::now().timestamp_millis();
    let last = *self.last_active.lock().unwrap();
    (now - last) > 300_000 // 5 minutes
}
```

1. Every detection cycle, the `last_active` timestamp is updated with the current time
2. On each cycle, before processing a new context, the system checks `is_idle()`
3. If `now - last_active > 5 minutes`, the context is overridden to `Idle` (0 Hz, fade out)
4. When activity resumes (user moves mouse, switches windows), `mark_active()` is called and the beat resumes

### What Counts as "Activity"

- **Active window change** — detected every detection cycle (3s default)
- **Window title change** within the same app — detected as part of window polling
- **Browser URL change** (via WebSocket from extension) — triggers a detection cycle

The idle detection does **not** use mouse movement, keyboard input, or system idle time APIs. It relies solely on window state changes.

## Auto-Pause Conditions

TuneSoar automatically pauses binaural beats (sets frequency to 0 Hz) under these conditions:

### 1. Meeting Detection

**Trigger:** Active window matches a meeting app (Zoom, Meet, Teams, Skype, FaceTime, Webex, etc.)

**Behavior:**
- Beat frequency set to 0 Hz
- Audio engine calls `fade_out()` (2-second fade)
- UI shows "Paused for Audio" with 📞 Meeting context

**Rationale:** Binaural beats would interfere with voice communication. Users need clear audio during calls.

### 2. Music Detection

**Trigger:** Active window matches a music app (Spotify, Apple Music, Tidal, etc.) **OR** YouTube URL contains audio keywords (`music`, `asmr`, `meditation`, `relaxing`, `sleep`, `lofi`, `ambient`)

**Behavior:**
- Beat frequency set to 0 Hz
- Audio engine calls `fade_out()` (2-second fade)
- UI shows "Paused for Audio" with 🎵 Music context

**Rationale:** Binaural beats would interfere with music listening. The user chose to listen to something else — respect that.

### 3. Idle Detection

**Trigger:** No window changes for 5 continuous minutes.

**Behavior:**
- Beat frequency set to 0 Hz
- Audio engine calls `fade_out()` (2-second fade)
- UI shows "Fading Out" with 💤 Idle context

**Rationale:** If the user walked away, don't play binaural beats into an empty room. Saves audio resources and prevents prolonged unconscious exposure.

### 4. Manual Pause

**Trigger:** User clicks the Pause button in the tray window UI.

**Behavior:**
- `is_paused` flag set to true
- Audio engine calls `fade_out()`
- UI shows pause state

**Note:** Manual pause is different from auto-pause. Manually pausing keeps the current context/beat selected; auto-pause overrides the context to Meeting/Music/Idle.

### Auto-Pause Summary Table

| Condition | Detection Method | Context Override | Freq | Fade | UI Label | Auto-Resume? |
|---|---|---|---|---|---|---|
| Meeting app | App name match | `Meeting` | 0 Hz | 2s fade out | "Paused for Audio" | Yes (when meeting app closes) |
| Music app | App name match | `Music` | 0 Hz | 2s fade out | "Paused for Audio" | Yes (when music app closes) |
| YouTube audio | URL keyword match | `Music` | 0 Hz | 2s fade out | "Paused for Audio" | Yes (when URL changes) |
| Idle (5 min) | Timestamp diff | `Idle` | 0 Hz | 2s fade out | "Fading Out" | Yes (on next window change) |
| Manual pause | User button click | None | N/A | 2s fade out | Paused | No (manual resume only) |

## Time-Based Fallback (Sleep Mode)

When no app or URL match is found, TuneSoar checks the local system time:

| Time Range | Behavior |
|---|---|
| **22:00 – 05:59** | If window title contains keywords (`kindle`, `reader`, `books`, `pocket`, `instapaper`) → **SleepPrep** (Delta 2 Hz). Otherwise → **Ambient** (Alpha 10 Hz). |
| **06:00 – 21:59** | → **Ambient** (Alpha 10 Hz) |

This ensures that late-night use defaults to sleep-friendly frequencies.

## Detection Flow — Complete Walkthrough

Here's what happens in a typical detection cycle (every 3 seconds):

```
1. Poll active window (platform-specific)
   ├─ Windows: windows-rs (GetForegroundWindow + GetWindowText)
   ├─ macOS: core-foundation (NSWorkspace.sharedWorkspace)
   └─ Linux: x11rb (_NET_ACTIVE_WINDOW + _NET_WM_NAME)

2. Read browser URL (if WebSocket extension connected)
   └─ Stored in ContextState.browser_url (Arc<Mutex<Option<String>>>)

3. Check idle
   └─ if (now - last_active) > 300_000ms → override to Idle (0 Hz)

4. Run detector.detect(window_title, app_name, url)
   ├─ Try URL match first
   │   └─ Case-insensitive substring + YouTube special handling
   ├─ Try exact app name match
   │   └─ Lookup in app_mappings HashMap
   ├─ Try fuzzy app match
   │   └─ Case-insensitive contains on app_name and window_title
   └─ Time-based fallback
       ├─ 22:00–06:00 → Check reader/research keywords → SleepPrep or Ambient
       └─ Otherwise → Ambient

5. Update beat profile
   ├─ If freq == 0 → fade_out() (meeting, music, idle)
   ├─ If profile changed → set_profile() (crossfade)
   └─ If no engine exists → BinauralEngine::new()

6. Store context in current_context for UI polling

7. Update last_active timestamp
```
